01 — Context

The Clinical Problem

ICU mortality rates range from 10–29%. Deterioration events — septic shock, respiratory failure, cardiac arrest — are rarely sudden. They are preceded by hours of subtle physiological drift. The challenge is detecting this drift automatically, across all patients simultaneously, before it becomes irreversible.

⚠️

Why Manual Monitoring Fails

ICU nurses monitor 2–4 patients simultaneously. A patient's SpO2 trending down by 1% per hour over 8 hours is invisible to the human eye amid alarm noise — but it is a strong statistical signal a model can detect.

🎯

The Prediction Task

Given 12 hours of patient vital sign history, predict whether a clinically significant deterioration event (SpO2 < 90% or HR > 110 bpm) will occur in the 6-to-12 hour window following the observation.

🏥

Why a 6-Hour Safety Gap?

Real early warning systems are useless if they alert only seconds before an event. The 6-hour gap between observation and prediction window is intentional — it gives clinical staff actionable response time to intervene and prevent the event entirely.

02 — Data Source

MIMIC-IV: Data Ingestion & Cleaning

We use the MIMIC-IV Demo dataset from PhysioNet — a publicly available, de-identified subset of the MIMIC-IV clinical database from Beth Israel Deaconess Medical Center. It contains ICU stay records for 100 patients with full vital sign time series, demographics, and outcomes.

The Raw Data Structure

MIMIC-IV is organized into two schemas. We read from compressed .csv.gz files:

  • hosp/patients.csv.gz — Demographics: subject_id, gender, anchor_age
  • hosp/admissions.csv.gz — Hospital admissions with admission types
  • icu/icustays.csv.gz — ICU stays with intime, outtime, stay_id
  • icu/chartevents.csv.gz — Timestamped vital signs measurements. This is our primary table (~millions of rows in production, small in demo).
Key Design Choice: We filter chartevents to only include measurements taken within the patient's ICU stay window (charttime >= intime AND charttime <= outtime). This prevents pre-admission or post-discharge data from contaminating features.

MIMIC-IV Item IDs Mapped

In MIMIC-IV, each measurement type has a numeric itemid. These are the 5 we use:

item_map = { 220045: 'heart_rate', 220277: 'spo2', 220210: 'resp_rate', 223762: 'temp', 220052: 'mean_bp' }

These are the standard CareVue/MetaVision item IDs for adult ICU monitoring.

03 — Label Design

The Temporal Labeling Strategy

This is the most critical and nuanced design decision in the entire project. A naive approach would label a patient as "high risk" if they ever had a low SpO2 reading. We reject that approach entirely. Here is why, and what we do instead.

The Naive Approach (Wrong): Label a patient's entire stay as positive if they ever deteriorated. This creates massive target leakage — the model would see data from after the event and learn spurious patterns, not predictive ones. A model trained this way will fail completely in production.

The 3-Window Architecture

We use a sliding window approach that generates multiple labeled samples per patient stay. For each position in time, we define three non-overlapping windows:

① Observation Window

12 hours

What: The historical data fed as model input. We compute statistical features over this window.

Why 12h? Long enough to capture physiological trends, short enough to avoid confounding changes in treatment protocol.

② Safety Gap

6 hours

What: A deliberately empty buffer between observation end and prediction start.

Why? This is the clinical "action window". If a nurse receives the alert now, they have 6 hours to assess and intervene before the predicted event window opens. Without this gap, predictions are operationally useless.

③ Prediction Horizon

6 hours

What: The future window where we check if deterioration occurs.

Label = 1 if: SpO2 < 90% OR HR > 110 bpm is observed at any time in this window.

Sliding Window Implementation — src/features.py

src/features.py — generate_sliding_window_features()
# Parameters: obs_hours=12, gap_hours=6, pred_hours=6 for stay_id, group in df_vitals.groupby('stay_id'): group = group.sort_values('charttime') start_time = group['charttime'].min() end_time = group['charttime'].max() # Initialize cursor at end of first possible observation window current_time = start_time + timedelta(hours=obs_hours) while current_time + timedelta(hours=gap_hours + pred_hours) <= end_time: # ① OBSERVATION WINDOW: [current_time - 12h, current_time) obs_window = group[ (group['charttime'] >= current_time - timedelta(hours=obs_hours)) & (group['charttime'] < current_time) ] # ② SKIP 6h SAFETY GAP pred_start = current_time + timedelta(hours=gap_hours) # +6h pred_end = pred_start + timedelta(hours=pred_hours) # +12h # ③ PREDICTION WINDOW: [current_time + 6h, current_time + 12h) pred_window = group[ (group['charttime'] >= pred_start) & (group['charttime'] < pred_end) ] # Skip if either window is empty (not enough data yet) if obs_window.empty or pred_window.empty: current_time += timedelta(hours=4) continue # Binary label: SpO2 < 90 OR HR > 110 in the FUTURE window # This is the deterioration event definition based on clinical guidelines is_deteriorated = ( pred_window[(pred_window['vital_name'] == 'spo2') & (pred_window['valuenum'] < 90)].shape[0] > 0 ) or ( pred_window[(pred_window['vital_name'] == 'heart_rate') & (pred_window['valuenum'] > 110)].shape[0] > 0 ) features['target'] = int(is_deteriorated) # Slide the window by 4h to generate the next sample current_time += timedelta(hours=4)
Why slide by 4 hours? Each patient's stay generates multiple training samples (overlapping in time by 8/12h). The 4-hour stride strikes a balance: fine-grained enough to capture evolving risk, coarse enough to avoid near-identical adjacent samples that would inflate the training set with redundant data and artificially smooth loss curves.
04 — Feature Engineering

What the Model Actually Sees

For each 12-hour observation window, we extract 3 statistical features per vital sign: Mean, Standard Deviation, and Linear Trend (slope). That gives us 15 vital-derived features + 2 static demographics = 17 total features going into the model.

The 3 Feature Types Explained

μ Mean

The average value across all measurements in the 12h window. Captures the baseline state. A mean SpO2 of 93% tells the model this patient is already near the dangerous threshold.

mean = Σ(x_i) / n

Clinical significance: Absolute vital levels. A chronically hypoxic patient (SpO2 93%) is fundamentally different from a healthy one at 99%.

σ Standard Deviation

Measures variability within the window. High variability in BP can indicate hemodynamic instability. High variability in HR can signal arrhythmia or autonomic dysfunction.

std = √(Σ(x_i − μ)² / n)

Clinical significance: Instability. A stable mean with high std often precedes deterioration — the patient is oscillating before crashing.

β Trend (Slope)

A 1st-degree polynomial fit on the time-indexed measurements. The slope coefficient tells us whether the vital is rising, falling, or stable across the window.

np.polyfit(t_indices, values, deg=1)[0]

Clinical significance: Velocity of change. This is the most powerful feature. A falling SpO2 slope of −0.5%/hour over 12h is a critical predictor, even if the current value is still in normal range.

src/features.py — compute_risk_features() — Full Feature Extraction
def compute_risk_features(series): """Computes 3 statistical features for a single vital's time series.""" if series.empty: return {'mean': 0, 'std': 0, 'trend': 0} res = { # Feature 1: Mean — where the vital sits on average 'mean': series.mean(), # Feature 2: Std — how volatile the vital is 'std': series.std() if len(series) > 1 else 0, # Feature 3: Trend — the slope of a linear fit over the window # np.polyfit returns [slope, intercept] — we only want slope [0] # np.arange creates integer time indices: [0, 1, 2, ..., n-1] 'trend': np.polyfit(np.arange(len(series)), series.values, 1)[0] if len(series) > 1 else 0 } return res # Called inside the sliding window for each of the 5 vitals: for v in ['heart_rate', 'spo2', 'resp_rate', 'temp', 'mean_bp']: v_data = obs_window[obs_window['vital_name'] == v]['valuenum'] stats = compute_risk_features(v_data) for stat_name, val in stats.items(): features[f'{v}_{stat_name}'] = val # e.g. 'spo2_trend', 'heart_rate_mean'

05 — Model Training

XGBoost — Why & How

We use XGBoost (eXtreme Gradient Boosting) as our primary classifier. This choice is deliberate and justified by several factors specific to clinical tabular data.

Why XGBoost over Neural Networks?

With ~2,300 training samples, deep neural networks would massively overfit. XGBoost regularizes natively via max_depth, subsample, and learning_rate. It's also fully interpretable via SHAP — essential for clinical trust and regulatory compliance.

Why XGBoost over Random Forest?

Gradient boosting learns sequentially, correcting the errors of previous weak learners. On small, imbalanced medical datasets, this iterative correction typically outperforms the parallel nature of Random Forests, especially for the positive minority class (deterioration events).

Why Temporal Train/Test Split?

We sort by timestamp and split 80/20 chronologically — NOT randomly. Random splits would leak future patterns into training (a form of temporal data leakage). A chronological split simulates the real-world deployment scenario: training on historical data, predicting on future patients.

Hyperparameter Choices Explained

max_depth = 4

Limits tree depth. Deep trees memorize training data. With only ~2300 samples and 17 features, a depth of 4 allows meaningful interactions (up to 4-way) without overfitting to noise in a small clinical dataset.

learning_rate = 0.05

Also called eta. Each new tree's contribution is scaled by this factor. A small LR means we need more trees (150 vs. 50) but the final model is more robust and generalizes better to the held-out test set.

subsample = 0.8

Each tree is trained on a random 80% sample of the training rows. This is stochastic gradient boosting — the randomness prevents any single outlier patient from dominating multiple trees and reduces overfitting.

eval_metric = 'auc'

We optimize for ranking performance (AUC) not binary accuracy. AUC is threshold-agnostic and handles class imbalance better — in our dataset, positive events are less common, making AUC a more meaningful optimization target than raw accuracy or logloss.

06 — Explainability

SHAP: Why Did the Model Alert?

Black-box predictions are unacceptable in clinical environments. We use SHAP (SHapley Additive Explanations) to provide feature-level explanations for every prediction — both globally for model validation and locally for each patient alert.

How SHAP Works

SHAP values are grounded in cooperative game theory (Shapley values). For each prediction, SHAP calculates each feature's marginal contribution to the output by iterating over all possible subsets of features. The result: each feature gets a signed value indicating how much it pushed the prediction above or below the baseline.

Interpretation example: If spo2_trend = −0.4 and its SHAP value is +0.18, it means this falling oxygen trend is pushing the risk score up by 0.18 compared to a patient with a stable SpO2. A clinician can see exactly why the model fired.
src/explainability.py
import shap def generate_explanations(model_path, data_path): model = joblib.load(model_path) df = pd.read_csv(data_path) X = df.drop(columns=['stay_id', 'subject_id', 'target', 'timestamp']) # TreeExplainer: optimized for tree-based models # Returns exact Shapley values (not approximations) explainer = shap.Explainer(model, X) shap_values = explainer(X) # Global summary: which features matter most overall? shap.summary_plot(shap_values, X, show=False) plt.savefig("reports/shap_summary.png") # View: bar chart of mean |SHAP| per feature

07 — Production Monitoring

Distribution Drift Detection (KS-Test)

A model trained on historical data degrades as the real world changes — seasonal disease patterns, new admission types, updated clinical protocols. We monitor for this drift automatically using the Kolmogorov-Smirnov (KS) test.

The KS Test — What It Is

The KS test is a non-parametric statistical test that measures the maximum absolute difference between two empirical cumulative distribution functions (ECDFs). It answers the question: "Could these two samples have been drawn from the same distribution?"

D = max|F_training(x) − F_current(x)|

The test returns a statistic D and a p-value. A p-value below 0.05 means: "There is less than a 5% chance these distributions are the same — this feature has statistically drifted."

Why KS and not PSI? The Population Stability Index (PSI) requires binning and is sensitive to bin choice. KS is exact, bin-free, and valid for small sample sizes — which is important in a clinical context where we may have limited recent inference data.
src/monitoring.py — detect_drift()
from scipy.stats import ks_2samp def detect_drift(reference_path, current_path): ref_df = pd.read_csv(reference_path) cur_df = pd.read_csv(current_path) vitals = [ 'heart_rate_mean', 'spo2_mean', 'resp_rate_mean', 'temp_mean', 'mean_bp_mean' ] for v in vitals: # ks_2samp compares two independent samples stat, p_value = ks_2samp( ref_df[v].dropna(), cur_df[v].dropna() ) is_drift = p_value < 0.05 status = "[DRIFT]" if is_drift else "[OK]" print(f"{v:22} p={p_value:.4f} {status}")

08 — REST API

FastAPI: Inside the Inference Engine

The FastAPI backend is the runtime core of the system. It serves the dashboard UI, provides the prediction endpoint, and handles all real-time feature engineering. Here is exactly what happens for every prediction request.

Request Lifecycle — Step by Step

Step 1: Startup — Model Loading

When uvicorn starts, FastAPI loads models/icu_model.joblib and models/feature_columns.joblib into memory once. The feature columns list is critical — it defines the exact order and set of features the model expects. This ensures the API always builds its feature matrix in the same column order the model was trained on, even if only partial vitals are sent.

api/main.py — Model Loading at Startup
try: model = joblib.load("models/icu_model.joblib") feature_columns = joblib.load("models/feature_columns.joblib") except: model, feature_columns = None, None # graceful degradation

Step 2: Schema Validation (Pydantic)

The incoming JSON is parsed against the VitalInput Pydantic schema before any business logic runs. If a field is missing, has the wrong type, or is malformed, FastAPI immediately returns HTTP 422 with a detailed validation error. This prevents corrupt data from ever touching the model.

api/main.py — Request Schema
class VitalInput(BaseModel): subject_id: str # Patient identifier (de-identified) stay_id: str # ICU stay identifier gender: int # 1=Male, 0=Female (pre-encoded) anchor_age: int # Patient age vitals: Dict[str, List[float]] # e.g. {"heart_rate": [80, 85, 90, ...]} # Example valid request body: { "subject_id": "P-1001", "stay_id": "S-4521", "gender": 1, "anchor_age": 72, "vitals": { "heart_rate": [88, 89, 91, 94, 97], "spo2": [96, 95, 94, 93, 92], "resp_rate": [18, 19, 20, 21, 22], "temp": [37.2, 37.4, 37.6], "mean_bp": [78, 74, 70, 66] } }

Step 3: Transient Feature Engineering

The raw List[float] for each vital must be transformed into the same features the XGBoost model was trained on. This is done on-the-fly inside the /predict handler. We compute Mean, Std, and Trend for each vital from the submitted list — without needing to store patient history server-side.

api/main.py — Inline Feature Engineering
features = {'gender': data.gender, 'anchor_age': data.anchor_age} vital_names = ['heart_rate', 'spo2', 'resp_rate', 'temp', 'mean_bp'] for v in vital_names: values = data.vitals.get(v, []) # Default to empty if vital not sent if values: series = pd.Series(values) features[f'{v}_mean'] = series.mean() features[f'{v}_std'] = series.std() if len(series) > 1 else 0 features[f'{v}_trend'] = np.polyfit( np.arange(len(series)), series.values, 1 )[0] if len(series) > 1 else 0 else: # If a vital is missing, fill with zeros — maintains column schema features[f'{v}_mean'] = features[f'{v}_std'] = features[f'{v}_trend'] = 0 # Build DataFrame and align to exact training column order # reindex fills any missing columns with 0 — robust to partial vital submissions df_input = pd.DataFrame([features]).reindex(columns=feature_columns, fill_value=0)

Step 4: Inference & Risk Classification

predict_proba returns a 2-element array: [P(class=0), P(class=1)]. We take [:, 1] — the probability of deterioration. This continuous score is then classified into three actionable risk tiers.

api/main.py — Inference and Classification
# model.predict_proba returns shape (n_samples, 2) # [0] = first (only) sample, [1] = probability of class 1 (deterioration) risk_score = float(model.predict_proba(df_input)[0, 1]) # Three-tier clinical risk classification # Thresholds chosen to balance alarm burden with detection sensitivity if risk_score > 0.7: risk_level = "HIGH" # Immediate bedside assessment elif risk_score > 0.4: risk_level = "MODERATE" # Increased monitoring frequency else: risk_level = "LOW" # Routine monitoring continues return { "patient_id": data.subject_id, "risk_score": round(risk_score, 4), # e.g. 0.8241 "risk_level": risk_level, # "HIGH", "MODERATE", or "LOW" "prediction_horizon": "6-12h", # always 6-12h in this version "model_version": "v1.0", "timestamp": datetime.now().isoformat() }

The Testing Strategy — 8 Tests, 3 Layers

The test suite is organized into three independent modules, each validating a different layer of the system. All 8 tests pass. Here is what each one checks and why it matters.

Layer 1: Data Integrity — tests/test_data_validation.py

tests/test_data_validation.py — 3 tests
# TEST 1: No temporal data leakage # Verifies that the 'timestamp' column exists and has no nulls. # A null timestamp would make temporal ordering undefined — a leakage risk. def test_data_leakage(): df = pd.read_csv(DATA_PATH) assert 'timestamp' in df.columns # column must exist assert not df['timestamp'].isnull().any() # no missing timestamps # TEST 2: No null patient/stay identifiers # A null stay_id would corrupt the train/test grouping — we'd assign a sample # to no patient, making temporal split and drift monitoring meaningless. def test_no_null_critical_ids(): df = pd.read_csv(DATA_PATH) assert not df['stay_id'].isnull().any() assert not df['subject_id'].isnull().any() # TEST 3: Binary target integrity # Confirms the target column only contains {0, 1}. # Any other value (e.g., NaN, 2, -1) would silently corrupt the classifier. def test_target_distribution(): df = pd.read_csv(DATA_PATH) assert set(df['target'].unique()).issubset({0, 1})

# Build the image docker build -t icu-monitor . # Run the container docker run -p 8000:8000 icu-monitor # Dashboard: http://localhost:8000 # API docs: http://localhost:8000/docs

Why Single-Container?

For this project scope, the FastAPI server handles both API and static file serving. In production at scale, you would separate these: an Nginx container for static assets and a separate uvicorn/gunicorn cluster for the API, behind a load balancer.

11 — Orchestration

The Pipeline Orchestrator

pipeline.py is the single entry point that runs all 5 stages in sequence. It uses Python's subprocess module to call each module independently, which means any stage failure halts the pipeline immediately with a clear error message.

pipeline.py — Full Orchestration Logic
def run_step(command, description): # subprocess.run with check=True: raises CalledProcessError if exit code != 0 # capture_output=True: captures stdout/stderr for logging result = subprocess.run(command, shell=True, check=True, capture_output=True, text=True) print(result.stdout) # On failure, prints stderr and calls sys.exit(1) — stops the pipeline def main(): # Stage 1: Ingest and clean MIMIC-IV data → data/vitals_cleaned.csv run_step("python src/data_processing.py", "Data Processing") # Stage 2: Sliding window feature engineering → data/features_final.csv run_step("python src/features.py", "Feature Engineering") # Stage 3: Train XGBoost, log to MLflow, save joblib artifacts run_step("python src/train.py", "Model Training") # Stage 4: KS-test drift check on training vs. current data run_step("python src/monitoring.py", "Drift Detection") # Stage 5: Run all 8 automated tests run_step("pytest tests/", "Automated Tests")
To run the full pipeline end-to-end: python pipeline.py from the project root. The output at the end should be: [SUCCESS] Full Pipeline Completed Successfully! If any stage fails, the error from that subprocess stderr will be printed and the pipeline halts — no partial states.
12 — Data Versioning

DVC: Tracking Data & Models Like Code

Raw data and model binaries cannot go in Git (file size limits, binary diffs). DVC (Data Version Control) extends Git to track these artifacts by storing metadata (hashes) in Git and pushing the actual files to a remote storage backend (S3, GCS, or local).

What DVC Tracks

  • data/vitals_cleaned.csv
  • data/features_final.csv
  • data/training_reference.csv
  • models/icu_model.joblib

Each file gets a .dvc pointer file committed to Git. The .dvc file contains the file hash and remote location — not the data itself.

Why This Matters for MLOps

If you change the feature engineering logic (e.g., add a 4th statistical feature), DVC lets you create a new version of features_final.csv and track it alongside the model version that was trained on it. You can reproduce any historical experiment by checking out a commit and running dvc pull.

Key Commands

# Initialize DVC in the repo dvc init # Track a data file dvc add data/vitals_cleaned.csv # Push data to remote storage dvc push # Pull data matching current Git commit dvc pull
